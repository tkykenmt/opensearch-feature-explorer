# Cross-Cluster Replication

## Summary

Two bug fixes for the Cross-Cluster Replication (CCR) plugin to restore compatibility with OpenSearch 3.6.0 after a core engine refactor. The core OpenSearch PR #20585 extracted `IndexingStrategy` and `DeletionStrategy` from `InternalEngine` inner classes into top-level classes and introduced `IndexingStrategyPlanner` / `DeletionStrategyPlanner` interfaces, which broke CCR's `ReplicationEngine` in two ways: compilation failures and runtime assertion failures.

## Details

### What's New in v3.6.0

#### Fix 1: Typo Alignment in `validFileNameExcludingAsterisk` (PR #1639)

Core OpenSearch PR #20620 renamed the method `Strings.validFileNameExcludingAstrix` to `Strings.validFileNameExcludingAsterisk` (fixing a typo). CCR's `ValidationUtil.kt` called the old method name, causing a compilation failure. This fix updates the method call to match the corrected name.

Changed file:
- `src/main/kotlin/org/opensearch/replication/util/ValidationUtil.kt`

#### Fix 2: ReplicationEngine Compatibility with Strategy Planner Refactor (PR #1647)

Core OpenSearch PR #20585 refactored `InternalEngine` by:
1. Extracting `IndexingStrategy` and `DeletionStrategy` from inner static classes into top-level classes in `org.opensearch.index.engine`
2. Introducing `IndexingStrategyPlanner` / `DeletionStrategyPlanner` with an `OperationStrategyPlanner` interface that includes its own `assertNonPrimaryOrigin` check

This broke CCR's `ReplicationEngine` in two ways:

1. **Compilation failure**: `IndexingStrategy` and `DeletionStrategy` were no longer inner classes of `InternalEngine` and required explicit imports
2. **Runtime assertion failure**: CCR replays operations with `Origin.PRIMARY` (via `TransportReplayChangesAction.performOnPrimary`) but routes them through `planIndexingAsNonPrimary` / `planDeletionAsNonPrimary`. The new planner's `assertNonPrimaryOrigin` check (separate from the engine-level override) rejected these operations

The fix modifies `ReplicationEngine` to construct lightweight copies of `Index` and `Delete` operations with `Origin.REPLICA` before passing them to the non-primary planning path:

```kotlin
override fun indexingStrategyForOperation(index: Index): IndexingStrategy {
    val replicaIndex = Index(
        index.uid(), index.parsedDoc(), index.seqNo(), index.primaryTerm(),
        index.version(), null, Operation.Origin.REPLICA,
        index.startTime(), index.getAutoGeneratedIdTimestamp(), index.isRetry,
        SequenceNumbers.UNASSIGNED_SEQ_NO, 0
    )
    return planIndexingAsNonPrimary(replicaIndex)
}

override fun deletionStrategyForOperation(delete: Delete): DeletionStrategy {
    val replicaDelete = Delete(
        delete.id(), delete.uid(), delete.seqNo(), delete.primaryTerm(),
        delete.version(), null, Operation.Origin.REPLICA,
        delete.startTime(), SequenceNumbers.UNASSIGNED_SEQ_NO, 0
    )
    return planDeletionAsNonPrimary(replicaDelete)
}
```

The copies use `null` for `versionType` and unassigned values for `ifSeqNo` / `ifPrimaryTerm`, as the `Engine$Index` and `Engine$Delete` constructors enforce these constraints for non-PRIMARY origins. This is safe because the non-primary planning path resolves conflicts using sequence numbers, not version types or conditional (optimistic concurrency) parameters. The actual indexing/deletion still operates on the original operation objects.

Changed file:
- `src/main/kotlin/org/opensearch/replication/ReplicationEngine.kt`

## Limitations

- The lightweight copy approach means strategy planning sees a different operation object than the actual indexing/deletion, which could diverge if future planner logic depends on fields zeroed out in the copy (e.g., `ifSeqNo`, `ifPrimaryTerm`)
